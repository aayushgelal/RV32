Introduction

Why RISC-V? Why Pipelining?

RISC-V: The Open Standard ISA

  • Open & free — no licensing fees, unlike ARM or x86
  • Modular design — base integer set (RV32I) + optional extensions
  • Industry adoption — used by SiFive, Google, NVIDIA, Qualcomm
  • Academic standard — clean ISA ideal for teaching architecture

Why Pipelining?

  • Throughput boost — overlap execution of multiple instructions
  • Higher clock speed — shorter critical path per stage
  • Real-world relevance — all modern CPUs are pipelined

Project Objectives

  • Implement full RV32I base instruction set (37 instructions)
  • Build a 5-stage pipeline with forwarding & hazard detection
  • Evolve from single-cycle → pipelined architecture
  • Add performance counters for quantitative analysis
  • Benchmark with real programs and compare CPI
  • Verify with comprehensive testbenches
Key Insight: A 5-stage pipeline can achieve ~4.4x throughput improvement over single-cycle, despite a ~14% CPI penalty from hazards.

Architecture

Single-Cycle Baseline

How Single-Cycle Works

  • Each instruction completes in one clock cycle
  • All hardware (ALU, memory, registers) used sequentially in one cycle
  • Clock period = longest instruction delay path
  • CPI is always 1.0 by definition

The Problem

Critical path bottleneck: The clock must be slow enough for the slowest instruction (typically LW: PC → IMEM → RegFile → ALU → DMEM → RegFile). Every instruction pays this penalty, even simple ones like ADD.

Single-Cycle Datapath

All components used combinationally in a single long cycle

Why Not Perfect 5x?

  • Pipeline hazards introduce stalls and flushes
  • Data hazards: instruction depends on result not yet available
  • Control hazards: branch decision not known until EX stage
  • Pipeline fill/drain: startup & shutdown overhead
Effective speedup = N / CPI where N = pipeline depth (5).
With CPI = 1.14: speedup = 5 / 1.14 = 4.39x

Hazards

Pipeline Hazards Overview

Data Hazards (RAW)

An instruction needs a register value that a previous instruction hasn't written back yet.

ADD x1, x2, x3 // writes x1 SUB x4, x1, x5 // reads x1 (not ready!)

Solution: Data forwarding + load-use stall

Control Hazards

Branch outcome unknown until EX stage — wrong instructions may have entered pipeline.

BEQ x1, x2, target // next 2 instrs already fetched // must be flushed if taken

Solution: Pipeline flush (2-cycle penalty)

Load-Use Hazard

Special case: load result only available after MEM stage, but next instruction needs it in EX.

LW x6, 0(x0) // data at MEM ADDI x7, x6, 1 // needs x6 in EX // cannot forward: 1-cycle stall

Solution: 1-cycle stall + MEM/WB forward

Hazard Resolution Strategy

Hazard TypeDetectionResolutionPenalty
RAW (1-gap) EX/MEM.rd == ID/EX.rs1/rs2 Forward from EX/MEM 0 cycles
RAW (2-gap) MEM/WB.rd == ID/EX.rs1/rs2 Forward from MEM/WB 0 cycles
RAW (3-gap) WB.rd == ID.rs1/rs2 same cycle Register write-through 0 cycles
Load-Use ID/EX.memRead && rd match Stall 1 + forward 1 cycle
Branch taken Branch resolved in EX Flush IF/ID + ID/EX 2 cycles
4 out of 5 hazard types resolved with ZERO penalty thanks to the forwarding unit. Only load-use requires a 1-cycle stall, and branches cost 2 cycles when taken.

Forwarding Paths

Forwarding Control Logic

// Forward A (ALU src1) assign forward_a = // EX/MEM hazard (1-gap) (ex_mem_reg_write && ex_mem_rd != 0 && ex_mem_rd == id_ex_rs1) ? 2'b10 : // MEM/WB hazard (2-gap) (mem_wb_reg_write && mem_wb_rd != 0 && mem_wb_rd == id_ex_rs1) ? 2'b01 : // No hazard 2'b00; // Forward B (ALU src2) — same logic // with id_ex_rs2 instead

Write-Through Bypass (3-gap)

// In regfile.v — handles WB→ID assign rd1 = (a1 == 0) ? 32'b0 : (we3 && a3 == a1 && a3 != 5'b0) ? wd3 : rf[a1];

Bug fix: without this, a 3-instruction gap would read stale register data.

Instrumentation

Performance Counters & Halt Detection

Hardware Performance Monitors

CounterWhat It MeasuresHow
perf_cycles Total clock cycles elapsed Increments every posedge clk
perf_instrs Retired (completed) instructions Counts when mem_wb_valid == 1
perf_stalls Load-use hazard stall cycles Counts when stall signal asserted
perf_flushes Branch/jump flush events Counts when flush signal asserted

Derived Metrics

CPI = perf_cycles / perf_instrs
Stall Rate = perf_stalls / perf_cycles × 100%
Flush Rate = perf_flushes / perf_cycles × 100%
IPC = 1 / CPI (instructions per cycle)
Speedup = N / CPI (vs single-cycle at same freq)

Valid Bit Propagation

To accurately count retired instructions, a valid bit propagates through all pipeline stages:

// Valid bit pipeline reg id_ex_valid, ex_mem_valid, mem_wb_valid; // Set valid = 1 when real instr enters ID // Set valid = 0 on stall bubble or flush // Propagate through EX → MEM → WB // Count only when instruction reaches WB if (mem_wb_valid && !halted) perf_instrs <= perf_instrs + 1;

Halt Detection

// Detect jal x0, 0 (self-loop) wire halt_instr = (if_id_instr[6:0] == 7'b1101111) // JAL && (if_id_instr[11:7] == 5'b0) // rd=x0 && (imm == 0); // offset=0 // When halted: freeze all counters if (halt_instr) halted <= 1;

Programs end with jal x0, 0 to halt cleanly. Counters freeze so results reflect only program execution.

ISA Extension

Custom ROL Instruction — ChaCha20 Quarter-Round

What is ChaCha20?

  • Stream cipher used in TLS 1.3, WireGuard, SSH, Android disk encryption
  • Core operation: the quarter-round — built entirely from ADD, XOR, and Rotate-Left
  • Standard RV32I has no rotate instruction — must emulate with 3 instructions (SLL + SRL + OR)
  • RISC-V Zbkb extension adds ROL/ROR for exactly this reason

Our Custom Instruction

// Added to alu.v 4'b1010: result = (src1 << src2[4:0]) | (src1 >> (32 - src2[4:0])); // ROL // Added to control.v 7'b0001011: begin // custom-0 opcode reg_write = 1; alu_src = 0; alu_control = 4'b1010; // ROL end

Uses RISC-V reserved custom-0 opcode space (0001011). R-type format, fully compatible with existing pipeline & forwarding.

ChaCha20 Quarter-Round Algorithm

// Input: a=0x61 b=0x62 c=0x63 d=0x64 // Repeat 4 rounds of: a += b; d ^= a; d <<<= 16; c += d; b ^= c; b <<<= 12; a += b; d ^= a; d <<<= 8; c += d; b ^= c; b <<<= 7; // Output (after 4 rounds): // a=0xD576E19B b=0x1898E8F8 // c=0x36858953 d=0x9FF722DF

Software (3 instrs per rotate) vs Hardware (1 custom ROL)

Standard RV32I — rotate d left by 16:

SLLI x5, x4, 16 // t0 = d << 16 SRLI x6, x4, 16 // t1 = d >> 16 OR x4, x5, x6 // d = t0 | t1

With Custom ROL — same operation:

ROL x4, x4, x12 // d = ROL(d, 16) // x12 holds rotation amount (16) // 1 instruction, 1 cycle, done.

Real-World Impact

  • IoT devices: Encrypt sensor data on slow MCUs without draining battery
  • Secure boot: Verify firmware signatures faster at startup
  • TLS 1.3: ChaCha20-Poly1305 is the default cipher for mobile & embedded
  • 26% fewer cycles per encryption block means proportionally lower energy per packet
Hardware cost: 1 extra ALU case (1 line of Verilog) + 1 opcode decode entry. Minimal area overhead for 1.35x crypto speedup.
Conclusion

Summary & Future Work

What We Built

  • Complete RV32I processor with 37 + 1 custom instruction
  • 5-stage pipeline (IF/ID/EX/MEM/WB) in Verilog
  • 9 hardware modules, 674 lines of synthesizable RTL
  • Full forwarding unit with 3 bypass paths
  • Custom ROL instruction for crypto acceleration
  • 5 testbenches with comprehensive verification

Key Results

  • CPI = 1.14 on Fibonacci (close to ideal 1.0)
  • 4.4x throughput over single-cycle baseline
  • 0 data stalls — forwarding handles all RAW hazards
  • 87.7% pipeline efficiency
  • 1.35x crypto speedup with custom ROL instruction
  • 26% cycle reduction on ChaCha20 quarter-round

Key Takeaways

  • Forwarding is essential — without it CPI would be 1.6–2.0x
  • Branch penalties are the dominant overhead in pipelined designs
  • Performance counters are critical for quantitative analysis
  • Pipeline design requires careful hazard analysis across all instruction gaps

Future Enhancements

  • Branch predictor — reduce 2-cycle penalty
  • Cache hierarchy — L1 I-cache + D-cache
  • RV32M extension — multiply/divide
  • More Zbkb crypto ops — ROR, ANDN, pack
  • FPGA synthesis — deploy on real hardware
  • Full ChaCha20 block — 20 rounds, 256-bit key
